OpenAI Responses 缓存兼容配置指南

本文说明 promptCacheKey、promptCacheRewriteLimit 和 codexTransportCompat 的作用、配置方法和适用场景。它们只影响 protocol: "openai-responses" 的模型。

什么时候需要配置

一般只在第三方 OpenAI Responses 兼容网关、中转站或自建代理上需要这些字段。官方 OpenAI endpoint 通常不需要。

场景 建议
普通 OpenAI Responses API 不配置这些字段，保持默认行为。
第三方网关要求 body 里有 prompt_cache_key 才能稳定命中缓存 配置 promptCacheKey: true。
第三方网关像 Codex endpoint 一样读取 x-codex-* header 配置 codexTransportCompat: true；如果它还要求 body key，同时配置 promptCacheKey。
服务端会返回自己的 prompt_cache_key 保持默认 promptCacheRewriteLimit: 3，让 Axiomate 自动复用服务端 key。
服务端每次都会改写 key，但你仍希望一直发送最新 key 设置 promptCacheRewriteLimit: 0。

这些字段不会保证 provider 一定命中 KV cache。它们只是按兼容网关常见约定发送稳定的请求标识；真正是否命中仍取决于网关和模型服务如何实现缓存。

字段速览

字段 类型 默认 作用
promptCacheKey boolean | string 未配置 控制 Responses 请求 body 是否发送专用 prompt_cache_key。
promptCacheRewriteLimit number 3 服务端连续改写 prompt_cache_key 达到多少次后停止发送专用 body key。0 表示永不因此禁用。
codexTransportCompat boolean 未配置 控制是否发送 Codex 风格 transport headers，例如 originator、session_id、x-client-request-id、x-codex-window-id 和 x-codex-turn-metadata。
userAgent string 未配置 已有字段。开启 codexTransportCompat 时优先作为 User-Agent；未配置时使用 Codex 兼容默认 UA。

promptCacheKey 怎么用

不开启

未配置或设为 false 时，Axiomate 不会发送专用 body prompt_cache_key，也不会记录服务端返回 key。

{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://api.example.com/v1",
      "apiKey": "sk-..."
    }
  }
}

使用默认 key

设为 true 时，请求 body 会发送默认 key：

a:{projectHash}:{providerHash}:{sessionHash}

实际请求中会渲染成类似：

a:9f3a1c2d4e5f6780:4a7b8c9d0e1f2345:5c9d2a71e8b0f3c4
{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://api.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true
    }
  }
}

默认 key 绑定 session，但使用短 hash 格式，长度为 52，适合要求 prompt_cache_key 不超过 64 字符的 Responses 兼容网关。

自定义模板

设为字符串时，Axiomate 会把它当作模板渲染。支持变量：

  • {projectHash}：当前项目路径 hash。
  • {providerHash}：当前 Responses provider 的 baseUrl hash。
  • {sessionHash}：当前 Axiomate session id 的 16 位 hash，默认模板使用它。
{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://api.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": "axiomate:project:{projectHash}:{providerHash}"
    }
  }
}

上面这个模板不带 {sessionHash}，会让同一项目、同一 provider 下的不同 session 使用同一个 client key。只有在你明确希望跨 session 复用时才建议这样配置。

很多网关要求 prompt_cache_key 最长 64 字符。模板变量只提供短 hash，不提供完整 session id。自定义模板仍应避免过长固定前缀。

和 extraParams.prompt_cache_key 的关系

extraParams.prompt_cache_key 仍然会作为普通透传字段发送，但不会启用服务端 key 状态机。

如果同时配置了专用 promptCacheKey 和 extraParams.prompt_cache_key，专用 promptCacheKey 会覆盖 body 里的 prompt_cache_key。

服务端 rewrite 行为

有些 Responses 兼容网关会在响应里返回自己的 prompt_cache_key。启用专用 promptCacheKey 后，Axiomate 会观察这个返回值。

  1. 如果返回 key 和请求 key 相同：认为稳定，连续 rewrite 计数清零。
  2. 如果返回了不同 key：保存服务端 key，下次请求改用服务端 key。
  3. 如果服务端连续多次返回不同于请求值的新 key：计数达到 promptCacheRewriteLimit 后，标记该 key 状态 unsupported，后续停止发送专用 body key。

promptCacheRewriteLimit 默认是 3。

配置 行为
promptCacheRewriteLimit: 3 默认。第 1、2 次连续 rewrite 继续复用新服务端 key；第 3 次连续 rewrite 后停止发送专用 key。
promptCacheRewriteLimit: 1 第一次 rewrite 就停止发送专用 key。
promptCacheRewriteLimit: 0 永不因 rewrite 禁用；服务端每次返回新 key，就保存并下次继续发送最新 key。

状态保存在 Axiomate 配置目录下的 prompt-cache-state 目录中。默认 key 带 {sessionHash}，因此默认状态也是 session 级隔离；自定义模板不带 {sessionHash} 时，状态会跨 session 复用。

codexTransportCompat 怎么用

设为 true 后，Axiomate 会在 OpenAI SDK request options 的 headers 中发送 Codex 风格 transport metadata。

{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "codexTransportCompat": true
    }
  }
}

只开启 codexTransportCompat 时，不会自动发送 body prompt_cache_key。这两个配置是正交的。

只开启 headers 时，header token 使用真实 Axiomate session id。会发送的 header 包括：

  • originator: codex_exec
  • session_id: <sessionId>
  • x-client-request-id: <sessionId>
  • x-codex-window-id: <sessionId>:0
  • x-codex-turn-metadata，内容包含 session_id、turn_id 和 sandbox: "none"。
  • User-Agent。如果 model 配置了 userAgent，优先使用它；否则使用 Codex 兼容默认 UA。

同时开启 promptCacheKey 和 codexTransportCompat

这时行为会尽量贴近 Codex/Hermes 兼容补丁：body prompt_cache_key 和 Codex headers 使用同一个 selected key。

{
  "models": {
    "main": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true,
      "promptCacheRewriteLimit": 3,
      "codexTransportCompat": true
    }
  }
}

如果服务端返回了新的 prompt_cache_key，后续 body 和 headers 都会改用服务端 key。如果达到 rewrite limit 并标记 unsupported，body key 和这些 compat headers 都会停止发送。

推荐配置模板

只解决 body cache key

{
  "models": {
    "responses-proxy": {
      "name": "Responses Proxy",
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true
    }
  }
}

Codex 风格中转站

{
  "models": {
    "codex-like-proxy": {
      "name": "Codex-like Responses Proxy",
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true,
      "promptCacheRewriteLimit": 3,
      "codexTransportCompat": true
    }
  }
}

需要指定 User-Agent 的中转站

{
  "models": {
    "codex-like-proxy": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": true,
      "codexTransportCompat": true,
      "userAgent": "codex_exec/0.120.0 (Ubuntu 22.4.0; x86_64) gnome-terminal (codex_exec; 0.120.0)"
    }
  }
}

如果没有特殊要求，不必手写 userAgent。开启 codexTransportCompat 且未配置 userAgent 时，Axiomate 会使用 Codex 兼容默认 UA。

跨 session 复用 client key

{
  "models": {
    "responses-proxy": {
      "protocol": "openai-responses",
      "model": "gpt-5",
      "baseUrl": "https://proxy.example.com/v1",
      "apiKey": "sk-...",
      "promptCacheKey": "axiomate:project:{projectHash}:{providerHash}"
    }
  }
}

这个模板不带 {sessionHash}，适合你明确希望同一项目、同一 provider 的多次会话共享同一个 cache namespace 的情况。

常见问题

现象 检查项
配置后没有看到缓存命中提升 确认模型是 openai-responses 协议；确认中转站确实读取 prompt_cache_key 或 Codex headers；确认 system prompt、工具列表、cwd、模型和 provider 没有频繁变化。
多个 Axiomate 实例互相影响 使用默认 promptCacheKey: true，它带 {sessionHash}。不要使用不带 session 的自定义模板。
想跨会话复用缓存 把 promptCacheKey 配成不带 {sessionHash} 的模板，例如 axiomate:project:{projectHash}:{providerHash}。
中转站每次都返回不同 key 保持默认 rewrite limit 会在连续 3 次 rewrite 后停止发送专用 key；如果你想一直跟随服务端 key，设置 promptCacheRewriteLimit: 0。
只想伪装 Codex header，不想发 body key 只设置 codexTransportCompat: true，不要设置 promptCacheKey。
只想发 body key，不想发 Codex header 只设置 promptCacheKey，不要设置 codexTransportCompat。
