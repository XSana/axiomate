Summary

结论摘要

Axiomate 不是一个“功能明显贫弱”的 agent 仓库。它在本地运行时、工具执行面、以及部分高价值子系统上已经具备真实实力，尤其是 本地执行、computer-use 与浏览器自动化深度 这类更贴近一线使用体验的工程面。

Hermes Agent 的领先点则主要不在“功能数量”本身，而在于把很多能力继续工程化成了更耐久的运行时：更稳定的任务闭环、更系统的记忆与上下文闭环、更强的恢复和跨入口连续性，以及更厚的平台化扩展面。

因此，最准确的判断不是“Axiomate 全面落后”，而是：Axiomate 在局部技术面有真实优势，但当前更大的差距集中在系统化运行时能力，而不是单点工具能力。

一句话判断： Axiomate 当前最需要补的不是盲目追平台表面积，而是优先补 持久化多代理编排与任务闭环，同时保住自己的强项，不要把本地执行和自动化深度稀释成泛平台跟随。
Axiomate 已具备多项强本地 agent 基础能力 Hermes Agent 的优势更集中在运行时耐久性与平台成熟度 最大误判风险：把局部强执行面误当成系统层已对齐 建议路线：先补持久化编排 / 记忆闭环 / 恢复能力，再决定平台扩张边界

Gap Analysis

详细差距分析

1. Durable Multi-Agent Orchestration

Hermes Agent 更接近“有调度、有租约、有恢复”的系统；Axiomate 当前更像高级协作原语集合。
高优先级

Hermes Agent 已有

  • Durable orchestration / worker lifecycle / task closure 语义更完整。
  • 更容易把多代理从 demo 级并发提升到长期运行系统。
  • release 与热度都在反复强化这条主线：multi_agent, provider_runtime, memory_context, ui_ux, plugins_tools。

Axiomate 现状

  • 通常已经有 team / swarm / message passing / backend 切换等 primitive。
  • 代表性问题不在“没有多代理”，而在“缺 durable board、lease、reclaim、ownership 级治理”。
  • Axiomate：agent, agent/.axiomate, agent/.axiomate/settings.local.json, agent/build.ts, agent/buildConfig.ts, agent/buildEnv.ts, agent/buildPaths.ts, agent/bunPluginComputerUseStub.ts

为什么重要

  • 这是多代理演示和多代理生产系统的分水岭。
  • 没有 worker lifecycle 治理，长任务一旦失败或漂移，就很难恢复和追责。
  • 这是 Hermes Agent 最值得优先学习的方法论，而不是只学表面 feature。
Axiomate：agent, agent/.axiomate, agent/.axiomate/settings.local.json, agent/build.ts, agent/buildConfig.ts, agent/buildEnv.ts, agent/buildPaths.ts, agent/bunPluginComputerUseStub.ts Hermes Agent：acp_registry/agent.json, agent, agent/account_usage.py, agent/anthropic_adapter.py, agent/auxiliary_client.py, agent/bedrock_adapter.py, agent/codex_responses_adapter.py, agent/context_compressor.py

2. Persistent Goals, Steering, Queueing

Hermes Agent 更像把长期目标保持和运行中纠偏提升成了一等控制原语；Axiomate 更依赖上下文稳定性。
高优先级

Hermes Agent

  • 更像具备跨 turn 的目标锚点，而不是只靠当前对话保持方向。
  • 更适合 queueing、handoff、多入口协同和中途 steer。

Axiomate

  • 即便已经有 task / agent / team primitive，也不等于已有 persistent goal runtime。
  • 当前更像把控制语义留给 prompt、todo 和记忆层承接。

差距含义

  • agent 一旦经历压缩、resume、handoff 或多代理分工，就更容易偏航。
  • 把 steer / goal 做成显式 runtime 语义，会显著提高可控性。

3. Self-Improvement Loop 与 Curator

Axiomate 常见的是 memory extraction；Hermes Agent 更像已经走到 review、curation、技能重构闭环。
高优先级

Hermes Agent 已有

  • review fork / curation / skills maintenance 的系统化程度更高。
  • memory、context compression、skill evolution 耦合更深。
  • 发布日志也明显把这条线当成主战略面。

Axiomate 已有

  • 记忆提取和会话沉淀说明基础并不弱。
  • 但更多是在“沉淀内容”，还不是“重构 agent 自身能力”。
  • Axiomate：agent/src/commands/context, agent/src/commands/memory, agent/src/commands/skills, agent/src/components/ContextSuggestions.tsx, agent/src/components/ContextVisualization.tsx, agent/src/components/memory, agent/src/components/PromptInput, agent/src/components/ShowInIDEPrompt.tsx

差距本质

  • 真正的差距不是“有没有 memory 功能”，而是“有没有 self-improving runtime”。
  • 如果 Axiomate 想进入更高阶 agent 竞争，这一块必须从 feature 升成闭环。

4. Runtime Durability：Checkpoint、Restart Resume、Long-Running Hygiene

Axiomate 能做 resume 不代表已经具备完整 runtime durability；Hermes Agent 更像把恢复做到了后台和系统层。
中高优先级

Hermes Agent

  • 更适合常驻进程、后台执行、gateway restart 后的恢复。
  • checkpoint hygiene、restart resume、长期运行保护更可能是一等能力。

Axiomate

  • resume / session restore 常常已经存在，说明这块不是空白。
  • 但通常更偏前台 transcript-based resume，而不是 runtime state resume。
  • Axiomate：agent/src/commands/resume, agent/src/components/ErrorBoundary.ts, agent/src/components/FallbackToolUseErrorMessage.tsx, agent/src/components/FallbackToolUseRejectedMessage.tsx, agent/src/components/ValidationErrorsList.tsx, agent/src/constants/errorIds.ts, agent/src/hooks/useNotifyAfterTimeout.ts, agent/src/hooks/useTimeout.ts

为什么这不是小差距

  • 如果未来想承载 cron、守护型 agent 或多代理后台协同，request-level resilience 不够。
  • 这块直接关系到系统是不是能长期跑，而不是只会一轮一轮对话。

5. Gateway 与跨平台会话连续性

这是最接近“产品边界差异”的差距：Hermes Agent 更像平台 runtime，Axiomate 更像 terminal-first / local-first agent。
高优先级

Hermes Agent 已形成的平台面

  • 跨入口会话连续性和 delivery 面通常更强。
  • 更像统一 agent runtime，而不是单一前端。

Axiomate 当前边界

  • 即便本地体验很强，也不等于已经进入 messaging / gateway / cross-channel 赛道。
  • 这块属于产品边界选择，不只是缺几个 adapter。

为什么要单独判断

  • 不能把这类平台差距误判成简单 feature parity。
  • 如果 Axiomate 不打算做平台，这里不必盲目追平；如果要做，就必须认真投入。

6. Plugin Surface 深度不足

Axiomate 更像已有插件分发与集成能力；Hermes Agent 更像把 runtime 行为表面本身做成可插拔。
中高优先级

Hermes Agent

  • plugin 不只是加工具，而是可以改 memory/provider/context/runtime 行为。
  • 跨入口复用与动态 hook 深度更高。

Axiomate

  • 已经有 plugin / MCP / skill 或 marketplace 相关基础，不该被说成没有生态。
  • 真正缺的是 hook surface 深度和平台内核可插拔程度。

这条备注必须写细

  • 用户说的“矩阵备注不够细”在这条最明显，因为“plugin 更强”这种写法太空。
  • 现在明确写成：分发与集成 != runtime surface 可插拔。

7. Execution Backends 与远程运行面

Axiomate 强在本机；Hermes Agent 更像在回答“agent 住在哪里都行”。
中高优先级

Hermes Agent

  • 更像拥有多后端或多入口运行面。
  • 这让 gateway、cron、background worker 脱离单机变得更可行。

Axiomate

  • 本地 shell、worktree、desktop / browser automation 更强时，不代表 remote runtime 也同样成熟。
  • 这块差距影响的是 agent 的部署形态，而不仅是功能 breadth。

为什么影响很大

  • 如果未来目标是平台化，远程执行后端会从“加分项”变成“必需品”。
  • 如果目标仍是本地执行型 agent，这条可以排在 durable orchestration 之后。

8. Session Search 与 Memory Infra 的成熟度差距

Axiomate 通常已经有可用 session search；Hermes Agent 更进一步把它做成 memory platform。
重点项

Hermes Agent

  • 更强调底层索引、provider 化、跨 session continuity。
  • 更容易与 curator、compression、external memory 打通。

Axiomate

  • session search 如果已经存在，就应该被明确承认为成熟基础，而不是被忽略。
  • 当前差距在 infra 深度和平台化，而不是有无功能。

判断口径

  • 这条在矩阵备注里要避免说成“都有 session search，所以差距小”。
  • 真正该写的是：一边是 usable subsystem，一边是 integrated memory runtime。

9. Cron / Unattended Automation 深度不足

Axiomate 即便已有 schedule / loop 原语，也未必已经形成 unattended automation runtime。
重点项

Hermes Agent

  • 更容易把 cron、delivery、resume、background runtime 连成系统。
  • 适合真正无人值守的 agent 工作流。

Axiomate

  • 有 cron primitive 只能说明起点不错。
  • 还需要 delivery、watchdog、restart hygiene、结果通知等能力配合。

为什么值得排进前十

  • 很多团队会高估“能 schedule”本身，而低估长期 unattended 运行的运维复杂度。

10. 写后自动校验与更强 guardrails

Axiomate 即便有 verify 类能力，也不代表已经把校验下沉到 write lifecycle；Hermes Agent 这方面更像默认 guardrail。
重点项

Hermes Agent

  • 把部分校验下沉到 write / patch 生命周期，可以显著抬高 agent 可靠性下限。
  • 这是 runtime guardrail，而不是“用户自己记得去 verify”。

Axiomate

  • 如果校验更多停留在 skill 或显式命令层，仍然说明 runtime 默认保护不够深。
  • 这类缺口不显眼，但非常影响实际交付稳定性。

为什么这件事该进入前十

  • 这条对 coding-agent 场景尤其关键，因为很多失败不是模型不会写，而是写后没人立即拦截。
Momentum

对方近期最重要的能力推进

Top 1. 持久化多代理看板

来源：Hermes Agent 最近 release / community / runtime 演进信号
高优先级

Hermes Agent 把多代理从“能拉起 worker”推进成“有看板、有心跳、有回收、有任务归属治理”的持久化运行时。

  • 这条线最值得直接学习，因为它不是多一个功能点，而是把多代理系统化。
  • 真正重要的不是 UI 看板本身，而是 worker 生命周期、重试预算、任务归属、回收等治理语义。

Top 2. 持久目标锁定与运行中纠偏

来源：Hermes Agent 最近 release / community / runtime 演进信号
高优先级

Hermes Agent 把 goal / steer / queueing 这类长期任务锚点提升成一等控制原语，而不是继续依赖 prompt discipline。

  • 这直接提升 agent 在压缩、resume、handoff、多代理分工时的抗偏航能力。
  • 如果维护侧也想做长任务运行时，这条能力应排在极前面。

Top 3. Gateway / 运行时重启恢复

来源：Hermes Agent 最近 release / community / runtime 演进信号
高优先级

Hermes Agent 近期明显强化了 checkpoint、restart recovery、long-running hygiene 这条线，让系统更接近后台可持续运行的运行时。

  • 这类能力的价值不在 demo，而在长期运行下的可靠性下限。
  • 对需要 cron、守护型 agent、跨入口 session continuity 的系统尤其关键。

Top 4. Autonomous Curator

来源：Hermes Agent 最近 release / community / runtime 演进信号
中高优先级

Hermes Agent 已经把 curator 推到前台：它不是记忆功能，而是 skill 库的持续审查、合并、归档与清理能力。

  • 这条线说明对方不只想让 agent 记住内容，而是让 agent 会维护自己的能力库。
  • 这正是维护侧应该直接抄的方法论，而不是只抄“记忆文件”表面。

Top 5. 升级后的 Self-Improvement Loop

来源：Hermes Agent 最近 release / community / runtime 演进信号
中高优先级

Hermes Agent 的 review fork / self-improvement loop 已经更系统：更像有 rubric、有优先级、有运行时继承、有约束边界的背景学习回路。

  • 这条线与 curator 共同构成“agent 会不会越跑越好”的关键。
  • 维护侧如果只做 memory extraction，不补 review loop，很快就会遇到上限。

Top 6. 新 TUI 与独立交互后端

Priorities

未来最该补的 10 件事

Priority

1. 给多代理补上 durable worker lifecycle

  • 差距所在：Axiomate 当前更像有 team / swarm / mailbox primitive，但缺少 durable board、worker lease、heartbeat、ownership、reclaim、zombie cleanup 这类 runtime 治理语义。
  • 直接抄 Hermes Agent 的 durable multi-agent orchestration 这整条线，尤其是 worker lifecycle、task ownership、retry budget、hallucination recovery、reclaim 机制，而不是只学“能 spawn 多个 agent”。
  • 为什么现在先补：这是多代理 demo 和多代理生产系统的最大分水岭；这块不补，后面的 cron、background tasks、长任务分工都会失控。
  • 落地动作：先做 task state machine，再补 heartbeat / lease / reclaim / timeout / ownership gate。
Priority

2. 把 persistent goal / steer 做成一等能力

  • 差距所在：Axiomate 现在更依赖上下文和 prompt 纪律保持方向，缺少跨 turn、跨压缩、跨 handoff 的显式目标锚点。
  • 直接抄 Hermes Agent 的 goal / steer / queueing 思路：把长期目标保持和运行中纠偏做成显式控制原语，而不是把它留给 prompt 和记忆文件自己兜底。
  • 为什么现在先补：没有 persistent goal，agent 一旦经历压缩、resume、多代理拆分或外部入口插话，就会更容易偏航。
  • 落地动作：先做 session-level goal pinning，再做 steer append，最后再做 queueing / deferred intent。
Priority

3. 从 memory extraction 走向 review loop

  • 差距所在：Axiomate 现在更像会沉淀 memory，但还不够会 review、merge、rewrite、promote skills，也就是还没形成 self-improvement loop。
  • 直接抄 Hermes Agent 的 curator / review fork / skill evolution 这条线，重点不是“记更多”，而是“定期审查、归档、更新 agent 自身能力”。
  • 为什么现在先补：这比继续堆 memory feature 更值钱，因为它决定 agent 会不会越跑越好，而不是越跑越乱。
  • 落地动作：先做 review report 和 skill update suggestion，再做 curator-lite，最后再考虑更完整的 curation pipeline。
Priority

4. 把 restart resume 升级成 runtime durability

  • 差距所在：Axiomate 即便已经能 resume transcript，也还不等于具备 checkpoint hygiene、background restart recovery、orphan task reclaim 这类 runtime durability。
  • 直接抄 Hermes Agent 的 checkpoint / restart recovery / long-running hygiene 思路，尤其是把恢复语义下沉到后台运行面，而不只是前台会话恢复。
  • 为什么现在先补：如果未来要支撑长任务、cron、守护型 agent 或多入口 runtime，这一层不补，系统会在重启和中断后持续丢状态。
  • 落地动作：把 resume 从 transcript restore 推向 runtime state restore，优先补 checkpoint pruning、restart recovery、orphan cleanup。
Priority

5. 下沉写后自动校验与 guardrails

  • 差距所在：Axiomate 当前即便有 verify 能力，也更偏显式调用；写后自动拦截、delta lint、runtime guardrail 仍不够深。
  • 直接抄 Hermes Agent 把部分校验下沉到 write / patch 生命周期的思路，直接学“默认保护”而不是学一个新命令。
  • 为什么现在先补：这类 guardrail 会直接抬高 coding-agent 的可靠性下限，比继续补外围工具更早见效。
  • 落地动作：先给 write / edit / patch 接 delta verifier，再逐步接最小 lint、结构检查和 failure surfacing。
Priority

6. 扩深 plugin runtime hook surface

  • 差距所在：Axiomate 现在更像已有 plugin 分发和集成，但 runtime 行为本身的 hook depth 还不够深。
  • 直接抄 Hermes Agent 那种 memory provider、context engine、model provider、tool transform、runtime hook 级别的可插拔思路，而不是只抄 marketplace 外壳。
  • 为什么现在先补：如果后面想扩 provider、memory、platform、dashboard 或 tool behavior，不先补 hook surface，系统很快会变硬编码。
  • 落地动作：优先开放 memory / context / provider / tool-result transform hooks，再考虑更大平台面。
Priority

7. 强化 task ownership 与 destructive-action safety

  • 差距所在：Axiomate 的工具越强，多代理和长任务场景下越需要明确 ownership、权限边界、回收策略和 destructive-action gate。
  • 直接抄 Hermes Agent 在多代理治理里对 ownership、reclaim、hallucination recovery、destructive tool control 的整体思路。
  • 为什么现在先补：如果没有 ownership 语义，多代理最容易在高权限工具和长任务里互相踩踏或重复破坏。
  • 落地动作：先给 destructive tools 加 ownership gate，再把 reclaim / retry / conflict policy 连起来。
Priority

8. 把 session search 升级成 memory infra

  • 差距所在：Axiomate 如果已经有 session search，这说明基础不错；真正差距在于它还没完全成为 integrated memory infra。
  • 直接抄 Hermes Agent 把 session recall、compression、retrieval、external memory、curation 串成一条 runtime 管线的做法。
  • 为什么现在先补：单独的 session search 很快就会触顶，只有升级成 memory infra，后续的 curator、steer、goal、handoff 才会更稳。
  • 落地动作：先补更结构化的 retrieval / compression 接口，再接外部 memory provider 和 review loop。
Priority

9. 把 cron / unattended automation 做成真正 runtime

  • 差距所在：Axiomate 即便已经有 cron / loop 原语，也未必已经具备 delivery、watchdog、timeout、restart-after-crash、result notification 这些 unattended runtime 语义。
  • 直接抄 Hermes Agent 把 cron、delivery、background runtime、no-agent watchdog 这类能力串起来的方式，而不是只抄 scheduler 表面。
  • 为什么现在先补：真正的无人值守运行会放大恢复、通知、超时与结果交付问题，所以这块必须按 runtime 来做。
  • 落地动作：先补 watchdog / timeout / notification，再补 crash recovery 和 background delivery。
Priority

10. 明确战略边界：做本地强执行，还是做平台 runtime

  • 差距所在：Axiomate 当前最危险的状态不是少几个 feature，而是产品方向摇摆：一边想追平台，一边又还没决定是否要承受平台复杂度。
  • 直接抄 Hermes Agent 的 lesson 不是“平台入口越多越好”，而是“一旦决定做平台，就必须把 gateway、backends、control plane、recovery 全部当内核工程来做”。
  • 为什么现在先补：如果战略边界不清，前 9 件事做起来都会变形，因为你不知道自己是在补本地执行型 agent，还是在造平台 runtime。
  • 落地动作：如果要进 Hermes Agent 赛道，就认真投入 gateway / remote backends / control plane；如果不进，就优先放大 local execution moat。

Axiomate 强项

  • 这类工程投入更接近核心产品能力，而不是附属小 feature。

Memory / Skill / 上下文工程

中高优先级

Axiomate 已有

  • memory / skills / context surface 已具备真实基础。
  • 上下文工程往往已经不是裸 prompt 拼接。

Hermes Agent 更强

  • retrieval、compression、curation、skill evolution 耦合更深。
  • 更接近 self-improving knowledge runtime。

评价

  • 真正该优先补的，不是“再加一个 memory feature”，而是把 Axiomate 的 memory / skill / context 做成闭环。
Roadmap

建议给维护侧的三阶段补强路线

Roadmap

Phase 1：把 Axiomate 变成更稳定的长任务 runtime

  • 补 durable worker lifecycle、persistent goal / steer、task closure 和 runtime recovery。
  • 把现有多代理和恢复能力从 primitive 升成系统语义。
Roadmap

Phase 2：把 memory 升级成 self-improvement loop

  • 在 memory extraction 之上做 review fork、skill update 建议、curation。
  • 建立可观测的 per-run learning artifacts，而不是只写 memory note。
Roadmap

Phase 3：明确是否进入 Hermes Agent 代表的平台赛道

  • 如果要进入平台赛道，就认真做 gateway / remote backend / control plane / plugin hooks。
  • 如果继续聚焦本地执行型 agent，就放大 computer-use、browser、verification、coding orchestration 的优势。
Strategy

战略判断：哪些该直接抄，哪些不必盲目追平

必须补

  • durable orchestration / worker lifecycle
  • persistent goals / steer / queueing semantics
  • self-improvement review loop
  • runtime durability / restart hygiene
  • 更深的 plugin runtime hooks
  • post-write verification / guardrails

不必盲目追平

  • 盲目追所有 platform adapter 数量
  • 在没有 clear strategy 前扩张完整 control plane
  • 只学 feature breadth，不学 lifecycle / recovery 方法
  • 为了平台叙事牺牲本地执行型优势
如果 Axiomate 的目标是继续做强本地执行型 agent，就应该优先复制 Hermes Agent 的治理方法而不是表面积；如果目标是做平台 runtime，则必须接受这是一条更重、更系统的工程路线。
Conclusion

最终判断

Hermes Agent 的强项不是“多几个工具”，而是把 agent 做成了更像 runtime 的系统：可恢复、可委派、可跨入口、可自我整理。

Axiomate 的强项则更集中在本地执行、CLI 体验、coding / runtime primitive，以及一线可感知的 automation 深度。两者不是完全同类，但如果比较 harness engineering 与 runtime maturity，Hermes Agent 当前更领先。

因此，最准确的结论不是“Axiomate 全面落后”，而是：Axiomate 在局部技术面有竞争力，甚至可能在本地执行上占优；但若要与 Hermes Agent 正面竞争更高阶 runtime / platform 能力，当前最需要优先补的是 durable orchestration and task closure。
