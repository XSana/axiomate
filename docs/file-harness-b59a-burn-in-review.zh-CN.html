结论

这轮不是只跑已有测试，而是按行为契约反查运行时代码和测试矩阵。结果发现并修复了两个实现偏差，均属于 false-positive / validation 一致性问题，没有发现会导致未读覆盖、stale 覆盖、path lock 死锁或 sibling 写入漏拦的高危问题。

审计方法

代码路径

  • FileReadTool：dedup、text partial、notebook read state。
  • FileEditTool：read-before-edit、partial edit、empty old_string、mtime/content fallback。
  • FileWriteTool：full-read overwrite guard、format envelope。
  • NotebookEditTool：read state content 口径、sibling、path lock。
  • BashTool._simulatedSedEdit：structured shell edit guard。
  • fileStateRegistry：same-process sibling、alias key、async path mutex。

检查方式

  • 从文档契约反查实现分支，重点看 validate 和 call 是否一致。
  • 优先补“应失败但当前放行”或“应允许但当前拒绝”的对抗性测试。
  • 对 path lock / sibling / partial-full 判定做 focused suite 回归。
  • 把任意 Bash/PowerShell 写入、跨进程 registry 继续标为已接受边界。

发现与修复

编号 问题 风险 修复 测试
BR1 Edit 对“已有空文件 + old_string: ""”在 validation 阶段直接放行。 不是误写风险，因为 execution 仍会拒绝未读已有文件；但 permission/validation 阶段和 call 阶段不一致，会让错误延后且绕开 fileHarnessFailure metadata。 已有空文件仍必须先经过 read/sibling/mtime guard；读过后允许用 Edit 填充空文件。 fileEdit.behavior.test.ts 新增未读空文件拒绝、已读空文件可填充。
BR2 Read notebook 时如果输入带 offset/limit，状态被记录成 partial。 Notebook Read 实际返回完整 cell view，不是文本 range read；误标 partial 会让后续 NotebookEdit 在 mtime-only drift 时 false stale。 Notebook Read 写入 readFileState 时固定记录为 full cell-view state：offset: 1、limit: undefined。 notebookEdit.behavior.test.ts 新增 offset/limit notebook read 后 mtime-only edit 可通过。
BR3 Edit partial read 后 mtime-only drift 的 execution 行为缺少主行为测试。 实现本身正确；缺少 call 级测试时，未来可能误把 partial read 当 full read 并写入。 未改逻辑，只补测试：partial read 后即使内容没变，只要 mtime drift，execution 也拒绝且不落盘。 fileEdit.behavior.test.ts 新增 partial mtime-only execution 拒绝。

覆盖矩阵

行为契约 当前状态 覆盖位置
Write 覆盖已有文件必须 full Read。 通过 fileWrite.behavior、failureMetadata
Edit 允许 partial Read 后精确替换，但 mtime/content/sibling stale 拒绝。 通过 fileEdit.behavior、failureMetadata
Full-read mtime-only drift 用 content fallback，partial read 不使用 fallback。 通过 fileEdit.behavior、fileWrite.behavior、notebookEdit.behavior
NotebookEdit 使用 FileReadTool 的 cell-view content 口径。 通过 notebookEdit.behavior
Same-process sibling writes 即使 mtime 恢复也要被拒绝。 通过 fileEdit.behavior、fileWrite.behavior、notebookEdit.behavior、fileStateRegistry
Path lock 对同一路径串行；same-chain reentry fail-fast，不死锁。 通过 fileStateRegistry、toolExecutionFileHarnessError
Symlink parent alias 进入同一个 registry key / lock key。 通过 fileStateRegistry、fileWrite.behavior、fileEdit.behavior
任意 shell / PowerShell 写入不进入 process-local registry。 接受边界 依赖 mtime/content/checkpoint；只有 internal _simulatedSedEdit 纳入 structured guard。
跨进程 registry 不做。 接受边界 跨 session 依赖 mtime/content 与 git checkpoint。

剩余风险

  • 硬链接 alias 不纳入当前 registry identity；如果 mtime 被恢复，仍依赖 content/checkpoint。
  • 任意 shell/PowerShell 写入不会记录 sibling registry；这是已接受的 harness 边界。
  • 跨进程/跨 CLI session 没有共享 registry；checkpoint 是跨 session 防线。
  • mtime 使用 floor 毫秒语义，降低误报，但同一时间粒度内的外部写仍主要靠 sibling registry 和 checkpoint。
