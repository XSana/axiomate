Hermes Evidence

Hermes 的“韧性/坚韧”版本系列给了什么参考

v0.7.0 - The resilience release

2026-04-03。核心不是单个重试函数，而是 provider、credential、gateway、compression、MCP 的系统性 hardening。

  • Same-provider credential pool：多 key 轮换、401 failover、429 时保留 pool 状态。
  • Per-turn primary runtime restoration：fallback 后下一轮恢复主 provider，并处理 transport recovery。
  • Anthropic long-context tier 429：不是盲 retry，而是降到 200k 并压缩。
  • Prevent compression death spiral：API disconnect 触发压缩失败循环时要有停止条件。
  • MCP stability fix pack：reload timeout、shutdown cleanup、OAuth non-blocking。

v0.13.0 - The Tenacity Release

2026-05-07。tag message 明确写着“Hermes Agent now finishes what it starts”。

  • Durable Kanban：heartbeat、reclaim、zombie detection、auto-block、per-task retries、hallucination recovery。
  • /goal：跨 turn 的 persistent Ralph loop，把“完成目标”做成 runtime primitive。
  • Checkpoints v2：单 store、真实 pruning、disk guardrails。
  • Gateway auto-resume：重启后恢复中断 session。
  • MCP：SSE transport、OAuth forwarding、stale-pipe retry、生命周期 keepalive。
  • Reliability closures：包括 hermes mcp add 没注册 server 却打开 chat 的回归。

v0.14.0 - 后续 API runtime 参考

2026-05-16。不是本次“坚韧”主证据，但对 Axiomate API 可靠性很有借鉴价值。

  • Codex app-server runtime：session reuse、wedged session retirement、post-tool watchdog、OAuth refresh classification。
  • Stream retry diagnostics：每次 drop 记录 inner cause、upstream headers、bytes、elapsed。
  • TTFB watchdog：无首包时主动中止，而不是等大 timeout。
  • 恢复 invalid_encrypted_content、null output、JSON decode、unsupported parameter 等真实错误形态。
  • quota failure 后 rotate pooled auth，避免烧同一无效 credential。

对 Axiomate 的启发：不要只照搬 Hermes 的功能面。真正应该复制的是“错误分类 - 恢复动作 - 观测证据 - 回归 fixture - 用户下一步”的闭环。

Part I

三类 API 请求链路的可靠性工程与重试

版本闸门

Axiomate 的三类协议已经通过 LLMProvider 抽象进入同一 orchestration 层：OpenAI Chat Completions、OpenAI Responses、Anthropic Messages。本文按协议分类讨论，但每类都要覆盖三种运行场景：主 agent 的 streaming request、stream-shape 失败后的 non-streaming fallback、side query / verify / compact 等非主循环请求。

协议链路 当前实现 主要风险 下一步判断
OpenAI Chat Completions
openai-chat		 openaiProvider.ts 构建 messages、max_tokens、tool schema、thinking vendor patch；stream 与 non-stream fallback 都走 withRetry。stream creation 的 404 fallback 已通过 deferModelNotFoundFallback 交给外层判定。 OpenAI-compatible provider 的错误形态很散：404 可能是 model not found，也可能是 stream endpoint 缺失；400 可能是参数不支持、stream 不支持、max_tokens 过大或 request validation；目前对 unsupported parameter、5xx validation、provider policy block 的分类还不够完整。 需补 contract matrix
OpenAI Responses
openai-responses		 openaiResponsesProvider.ts 使用 input、max_output_tokens、reasoning encrypted content round-trip；stream adapter 和 non-stream fallback 均已有。 Responses 的错误恢复比 Chat 更敏感：encrypted reasoning replay、null output、incomplete status、event order、function_call output 都会影响后续 turn。Hermes 已有 invalid_encrypted_content 和 null output 恢复案例，Axiomate 分类还没有覆盖这些语义。 优先补齐
Anthropic Messages
anthropic		 anthropicProvider.ts 通过注入的 buildParams 构造 params，保留 thinking signature、betas、non-streaming fallback、countTokens 与 verifyConnection。 Anthropic 的失败语义强绑定 thinking signature、long-context tier、OAuth/订阅能力、image payload、content block 顺序。Axiomate 已有 thinking_signature、long_context_tier，但缺 OAuth long-context beta、image too large、multimodal tool content downgrade 等分类。 已有基础，仍需扩表
API Reliability

Axiomate 当前可靠性基础

已经值得保留的设计

  • withRetry.ts 已经是中心 retry harness：默认 retry、jitter backoff、foreground source 限制、stale socket 处理、fallback model signal。
  • errorClassifier.ts 已经把 status code 与 message pattern 收束为结构化 reason，而不是到处写散落 string match。
  • llm.ts 已经把 stream unsupported / 404 endpoint / stream shape failure 与 transport/rate-limit/server-error 区分开，避免把 non-stream fallback 当万能恢复。
  • 近期提交连续修复了 semantic retry、OpenAI streaming 404 fallback、stream-shape fallback、max-tokens recovery，说明问题已经被定位到正确层级。

仍然不稳的原因

  • 分类表还不够覆盖真实 provider 错误。Hermes 的 classifier 已覆盖 invalid_encrypted_content、image_too_large、provider_policy_blocked、multimodal_tool_content_unsupported 等更细 reason。
  • 恢复动作没有形成统一 trace。用户和维护者很难一眼看到某次请求经历了 drop max tokens、disable thinking、stream retry、non-stream fallback、fallback model 或 reactive compact。
  • side query / inference 路径没有完全共享主循环的语义恢复。OpenAI/Responses 的 inference() 主要 wrap error，Anthropic 依赖 SDK retries，和 withRetry 的语义不完全一致。
  • 缺少跨协议 golden fixtures，导致每次修复容易只覆盖一个 provider 的具体 case。
Retry Contract

建议建立统一恢复动作表

失败类别 正确动作 不应该做 需要覆盖的协议
Transport / timeout 刷新 client / 断开 stale socket / jitter backoff / stream consumption retry；记录 elapsed、TTFB、bytes、request id。 不要自动切 non-streaming；这不是 stream shape incompatibility。 三类协议全部
Rate limit / overload 尊重 retry-after；foreground 才重试；连续 529 达阈值触发 fallback；credential pool 存在时先 rotation。 不要让 background summarizer、title、classifier 放大容量故障。 三类协议全部
Context / payload 区分 input overflow、max_tokens-too-large、413 payload；能调 output cap 就调 cap，不能就交给 reactive compact。 不要把所有 400 + max_tokens 都当 compact，也不要在压缩失败后继续死循环。 三类协议全部
Stream shape 仅当 endpoint 不支持 streaming、非 model-not-found 404、local stream event 顺序错误时进入 non-streaming fallback。 不要对已经开始 tool call 的 partial stream 直接 non-stream fallback，避免重复工具执行。 OpenAI Chat、Responses 优先
Reasoning / thinking replay Anthropic signature 错误时 disable thinking；Responses encrypted content 无效时 strip replay state 并重试。 不要把 replay 错误当普通 400 后反复失败。 Anthropic、OpenAI Responses
Unsupported request fields 精确识别 unsupported parameter / temperature / max token 字段，移除字段后只重试一次。 不要对所有 400 做 blind retry，也不要在 5xx request-validation 上无限 retry。 OpenAI-compatible 链路优先
Protocol Deep Dive

三类协议的具体修复建议

P0

OpenAI Chat

目标：把近期多轮修复沉淀成稳定 contract，而不是继续按 case patch。

  • 为 chat.completions.create(stream=true) 建立 400/404/429/5xx 错误 envelope fixtures，分别覆盖 stream unsupported、model_not_found、endpoint_not_found、unsupported parameter、max_tokens_too_large、rate limit、server error。
  • 把 deferModelNotFoundFallback 的行为写成测试矩阵：有 fallback model、无 fallback model、404 中包含 model name、404 不包含 model name。
  • 在 retry trace 中记录 request mutation：删除 max_tokens、删除 unsupported field、禁用 thinking、切换 non-streaming、切换 fallback model。
  • 把 non-streaming fallback 的触发条件只保留给 stream-shape incompatibility，并给 transport/rate-limit/server-error 加负向测试。
P0

OpenAI Responses

目标：把 Responses 的 reasoning replay、event stream、non-stream response 三种形态当成独立协议，不要只当 Chat 的字段变体。

  • 补 invalid_encrypted_content 分类：当 encrypted replay 被拒绝时，清理本轮 replay state，重试一次，并记录 trace。
  • 补 null output / empty content recovery：区分 legitimate empty、incomplete max_output_tokens、provider malformed response。
  • Responses stream adapter 建立 event-order golden：response.created、output_item.added、content_part.added、function_call、reasoning、incomplete。
  • 让 Responses 的 stream creation 404/fallback 行为与 Chat 对齐，避免 fallback model 抢先吞掉“stream endpoint 不支持”的判断。
P1

Anthropic

目标：保留现有 thinking 和 long-context 处理，同时补齐 Hermes 已踩过的真实 provider 失败形态。

  • 在 errorClassifier.ts 增加 oauth_long_context_beta_forbidden、image_too_large、multimodal_tool_content_unsupported。
  • 把 long-context tier 429 的恢复从“shouldCompress”升级为明确 action：降上下文 tier、更新压缩器状态、压缩后重试，并区分订阅 tier 与模型能力。
  • 补 thinking-only empty response、signature invalid、tool content block 顺序错误的 fixture。
  • 让 verify/countTokens/inference 路径的失败也能返回同一 taxonomy，至少能进入 doctor/failure card。
Test Matrix

测试必须从“单元测试”升级为“协议 contract”

Request body golden

同一 neutral intent 在三类协议下生成的 body 要有快照。特别是 tools、tool_choice、system、thinking、image、max token 字段。

Stream chunk golden

收集最小 stream chunk 序列，覆盖首包、usage、tool call、reasoning、partial、empty、malformed。

Error envelope golden

每个 provider 常见错误体都要映射到确定 reason，不能靠临时字符串 patch。

Retry trace golden

每次恢复动作都应生成结构化 trace：attempt、reason、mutation、delay、fallback、final outcome。

建议新增一个 api-contract 测试目录，按 openai-chat、openai-responses、anthropic 分组。规则是：凡是新增一个错误 pattern 或 retry action，必须同时新增 error fixture 和 retry trace fixture。
Part II

产品化修复：把能力变成可用体验

非功能堆叠

第二部分不讨论新能力扩张，而讨论用户实际遇到的摩擦：MCP 服务无法像 marketplace 那样添加源和验证；provider 配置要求用户理解太多协议字段；失败时缺少具体下一步；goal/checkpoint 等强能力需要更清楚地解释状态边界。

产品面 当前症状 目标状态 优先级
MCP 添加与源发现 CLI 能添加 server，/mcp 能管理已有连接，但用户心智需要“添加源、浏览、验证、启用”。 /mcp 内置 Add Server wizard、catalog/source、dry-run verify、失败卡片和一键编辑。 P0
Provider onboarding 用户需要理解 protocol、baseUrl、vendor template、context window、thinking、supportsImages、User-Agent。 普通模式只问 provider/API key/model；高级字段折叠；常见 provider preset 可验证。 P0
Doctor / Failure Card /doctor 有工程诊断能力，但失败时仍偏“看 debug log”。 按 API、MCP、plugin、settings 分类展示问题、根因、可执行 action。 P0
Goal / Checkpoint 解释 功能强，但用户需要知道当前 goal 是否 paused/resumed、checkpoint 是文件恢复还是会话恢复、失败后可回到哪里。 状态栏和命令输出以“当前状态 + 下一步动作”为中心，而不是暴露内部实现。 P1
概念统一 plugin marketplace source、plugin、MCP server、MCP catalog、model provider 都是“扩展”，但入口和文案不一致。 建立统一 extension mental model：来源、包、服务、凭据、启用状态、健康状态。 P1
Product Workstreams

建议的大版本工作包

P0

MCP Add / Catalog / Verify

要解决的问题：用户说“MCP 服务无法添加源”，不是只缺一个命令参数，而是 MCP 缺 marketplace 级别的产品面。

  • /mcp 增加 Add Server wizard：stdio、HTTP、SSE；command/URL、env、headers、OAuth、scope 分步骤输入。
  • 增加 MCP catalog/source 概念：官方推荐源、用户自定义源、本地 JSON 导入、来自 plugin 的 server。
  • 所有新增 server 必须 dry-run verify：能启动、能列 tools、能解析 schema、能展示失败 stderr/HTTP status。
  • 失败卡片提供 Retry、Edit、Disable、Copy Diagnostic、Open Config，不让用户只看到“失败，请看 debug”。
  • 复用 commands/mcp/addCommand.ts 和 services/mcp/config.ts 的写入逻辑，避免 TUI 与 CLI 两套语义。
P0

Provider Setup Presets

要解决的问题：Axiomate 的模型配置能力很强，但第一屏暴露太多协议细节。

  • 把常见 provider 做成 preset：OpenAI-compatible、Anthropic-compatible、OpenAI Responses、DashScope、SiliconFlow、DeepSeek、local endpoint。
  • 默认流程只问三件事：provider、API key、model；baseUrl 从 preset 来，contextWindow 通过 probe 或模板推断。
  • Advanced 面板再暴露 vendor template、thinking mapping、supportsImages、User-Agent、extraParams。
  • 配置保存前执行 minimal verify，并把失败映射到同一 failure card taxonomy。
P0

Doctor Action Center

要解决的问题：诊断不是列清单，而是把用户带到可修复状态。

  • /doctor 增加四类 failure card：API provider、MCP server、plugin/source、settings/schema。
  • 每张卡展示：影响范围、最后一次错误、配置来源、下一步 action、最小复现命令。
  • API failure card 使用第一部分的 retry trace，直接显示本轮为什么没有继续重试或为什么切了 fallback。
  • MCP failure card 展示 server command/URL、transport、timeout、stderr/HTTP 摘要、schema parse 错误。
P1

Goal / Checkpoint Product Layer

要解决的问题：goal 和 checkpoint 已经是 Axiomate 的强项，但需要从“命令集合”变成可理解的状态系统。

  • /goal status 输出只保留用户需要的：目标、子目标、预算、paused/running/done、下一步动作。
  • /checkpoints 明确区分 file restore、conversation restore、off-branch rows、orphan reachability。
  • 失败恢复路径要可解释：stream 失败、fallback model、compact、goal continuation、checkpoint anchor 之间谁拥有状态。
  • 保留高级诊断，但不要让普通用户先看 hash、internal row、debug-only 字段。
Release Plan

建议按三阶段推进

阶段 1：冻结协议面，补 contract

目标是让 API failure 不再靠临场 patch。

  • 建立三类协议 fixture：request body、stream chunk、error envelope、retry trace。
  • 扩展 classifier taxonomy，对齐 Hermes 已验证的真实错误类型。
  • 所有 retry action 进入结构化 trace。
  • side query / inference 至少能共享分类和 failure card。

阶段 2：修 MCP 与 onboarding 产品面

目标是新用户能完成配置，老用户能修失败。

  • /mcp Add Server wizard + catalog/source + dry-run verify。
  • Provider presets + Advanced 折叠。
  • /doctor failure cards + actionable fixes。
  • 统一 plugin/source/MCP/provider 的文案与配置入口。

阶段 3：端到端发版闸门

目标是把“能跑”变成“稳定可发布”。

  • 安装、provider setup、MCP 添加、主 agent 任务、goal、checkpoint、doctor 形成 E2E 脚本。
  • 失败路径验收：错误 key、错误 model、错误 MCP URL、stream unsupported、quota、context overflow。
  • 文档按任务重写：配置模型、添加 MCP、修失败、使用 goal、恢复 checkpoint。
  • 发版前必须过 API contract matrix 和 product smoke matrix。

不建议此版本做

避免把产品化版本变成新功能版本。

  • 不要新增大量 provider adapter 来掩盖现有三类协议不稳。
  • 不要复制 Hermes 的 messaging/platform breadth。
  • 不要把所有诊断都堆进 debug log。
  • 不要让 MCP catalog、plugin marketplace、provider preset 各自形成一套概念。

Final Evaluation

最终评估

Axiomate 下一大版本的核心命题不是追平 Hermes 的广度，而是把已经存在的本地 agent 能力做成稳定产品。最先该解决的是三类 API 协议链路的可靠性：每一种失败都应该有确定分类、确定恢复动作、确定观测证据和确定回归 fixture。其次才是产品化入口：MCP 添加、provider onboarding、doctor failure card、goal/checkpoint 状态解释。

Hermes 的 v0.7 resilience 和 v0.13 Tenacity 说明，可靠性不是 retry 次数，而是 runtime 生命周期治理。Axiomate 已经具备把这些治理落到 TypeScript 协议层和 TUI 产品层的基础。下一步要做的是收口，而不是扩张。
